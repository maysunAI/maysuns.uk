# 回答之前，先把请求拆解映射清楚

## 解决的问题

真实的请求很少是一个干净利落的问题。往往是好几件事混在一起——一个纠正、两个新要求、一个题外话、一段贴进来的背景资料——一口气打出来，想到哪写到哪。如果不加处理直接回答，AI助手往往会抓住最显眼的那部分，悄悄漏掉其余的。提出请求的人不一定会马上发现有东西被漏掉了，甚至可能一直都不会发现。

## 核心思路

在回答之前，先把收到的请求重新整理成一份编号列表，每一条都清楚标注它对应原文的哪一部分。只有这份映射整理好之后，才开始真正回答——按同样的顺序，逐条作答。

这不是为了文风好看而做的转述，而是一道结构性的检查关卡：在真正下笔回答之前，把"一段混杂着多个诉求的文字"变成"N条各自独立、可以逐一回答的条目"，这样就不会有任何一条被悄悄地漏答。

## 实际操作是什么样的

1. 先完整读完收到的整条消息，不急着回答。
2. 把它拆分成编号条目——纠正、新的请求、疑问、确认——有几条就是几条，不是固定数量。
3. 针对每一条明确作答，并注明它回应的是哪个编号。
4. 如果某一条确实需要更多信息才能回答，就把这一点明确列成单独的一条说清楚——不能直接跳过不提。

## 为什么重要——这也是「如何正确和AI说话」的方法

这个习惯不只是关于AI应该怎么回复，它是双向的。站在提问者的角度，实际做法是：当一个请求里有好几个部分时，主动把它们拆开（用编号列表、在不同诉求之间空一行，或者干脆停下来说"还有一件事"），而不是把五件不同的事塞进一段密密麻麻的文字里。输入的时候各个要点之间的界限越清楚，输出的时候就越不容易漏掉哪一条。

这件事恰恰在最容易出错的场景里最重要：长会话、一条消息里同时混杂着纠正和新工作、或者答案分好几批到达（后台Agent、多轮分批回复）——这些情况下，"是不是每一部分都真的被回应了"，已经不再是一眼就能看出来的事了。

## 跟T00其他部分的关系

这是T00的req/reply协议底下的那层纪律：持久化日志让请求和回答都能留存下来，而"映射"这个动作，正是防止日志里某一条密集的记录在被真正回答之前就悄悄丢失掉某些部分的关键。
