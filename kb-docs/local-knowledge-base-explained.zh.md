# 本地知识库（RAG + LangChain），解释一下

## 解决的问题

一个语言模型只知道它被训练过的内容。对于你的私人文档、公司内部Wiki，或者任何在其训练截止日期之后发布的内容，它并没有可靠的信息——除非你直接把这些材料交给它。每次文档一变就重新训练一遍模型是不现实的。本地知识库就是在不重新训练模型的前提下，让模型能够就*你自己的*特定内容给出准确、最新答案的方法。

## 核心技术：RAG（检索增强生成）

RAG不是单纯依赖模型的记忆，而是把模型和一个检索步骤配对使用：

1. 你的问题会通过**嵌入（embeddings，意义的数值表示）**做向量相似度搜索，与知识库进行比对——找到语义上相关的片段，而不仅仅是那些恰好共享完全相同关键词的片段。
2. 检索出最相关的片段。
3. 这些片段连同你的问题一起被喂给模型，这样生成答案时,模型面前有真实的源材料，而不是靠模糊的记忆。

因为答案是基于检索到的文本生成的，一个RAG系统往往可以引用它来自哪个段落——把「相信我」变成「这是出处，你可以自己核实」。这对减少事实性问题上的幻觉非常重要。

## 常见工具：LangChain

手工搭建一套RAG流水线，意味着要反复解决同一批「管道工程」问题——文档分块、管理向量存储、格式化检索到的上下文、处理对话记忆。LangChain是一个开源框架，把这些都打包成可复用的构建模块（文档加载器、文本分割器、向量存储集成、记忆模块），这样就不用每一层都从零写起。

一旦一条流水线有好几个活动部件（检索 → 推理 → 调用工具 → 响应），LangChain带来的开销就物有所值了。而对于一次简单的单轮「提示-响应」调用，直接调用模型API往往更简单——框架并不天然就更好，这是一种取舍。

## 其他常见工具：LlamaIndex 和 LangGraph

**LlamaIndex** 专注于问题的另一半——「把你的数据接进来、变得可检索」：连接数据源（文件、数据库、API），对它们分块、建索引，并高效查询。当你需要的主要就是检索本身、不需要一整套通用编排框架时，它是比LangChain更窄但通常更简单的选择。

**LangGraph**（由LangChain团队打造）解决的是另一个问题：当一个应用需要的不只是一条直线流水线——带分支的逻辑、循环、可以回头重新走某几步的多步骤流程、多个互相协作的Agent——一条简单的链就不够用了。LangGraph把整个应用建模成一个「带显式状态的步骤图」，让这类更复杂、带循环、或多Agent协作的流程，比一串函数调用更容易推理和调试。通常是在「单纯的RAG（检索→回答）已经不够，应用需要在检索之上真正做多步骤智能体行为」时才会用到它。

粗略地摆一下这四者的位置：RAG是*技术*本身，LlamaIndex专注于实现它时「数据接入/检索」的那一半，LangChain提供通用的*流水线*构建模块，LangGraph负责当这条流水线需要分支、循环或多个Agent协作（而不是一条直线）时的*编排*工作。

## 「本地」具体指什么

完全在自己的机器上运行这一切——用本地向量存储加上本地运行的模型（例如通过Ollama），而不是云端API——意味着你的文档永远不会离开你的设备。代价通常是：用能力和速度换取隐私和离线可用性，相比托管模型会有所妥协。

## 适用场景

- 「与你的文档对话」类工具（Wiki、PDF、代码库）
- 必须依据某个特定产品文档来回答的客服机器人
- 任何一种场景，其中针对*特定的、不断变化的*知识体系的准确性，比通用的流畅表达更重要

它并不是万能升级。对于不依赖外部知识的任务（创意写作、通用推理），检索只会增加开销,而没有多少实际好处。
