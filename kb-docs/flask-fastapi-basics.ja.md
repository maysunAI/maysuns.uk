# Flask & FastAPIの基本

## それらが何であるか

FlaskとFastAPIは、いずれもAPIやウェブバックエンドを構築するために使われる軽量なPythonウェブフレームワークだ。どちらもルート（URL）と、リクエストがそこに届いたときに何が起きるかを定義でき、大きくてオールインワンのフルスタックフレームワークのようなオーバーヘッドを伴わない——しかし、両者は異なる設計思想の上に構築されている。

## Flask

Flaskは意図的に最小限で、こだわりが少ない（unopinionated）。小さなコア——ルーティング、リクエスト／レスポンス処理——だけを提供し、それ以外のほぼすべて（データベースアクセス、認証、バリデーション）は、必要に応じて拡張機能や自分自身のコードを通じて追加するのはあなた次第だ。これによりFlaskは柔軟で扱いやすくなっている。裏側で起きている「魔法」があまりないため、小規模なサービス、プロトタイプ、スタック全体を完全にコントロールしたいプロジェクトで人気の選択肢になっている理由の一つだ。

## FastAPI

FastAPIは、いくつかの特定のアイデアを中心に構築された、より新しいフレームワークだ。

- **Pythonの型ヒントを唯一の情報源とする。** リクエスト／レスポンスデータがどんな形であるべきかを標準的なPythonの型アノテーションを使って宣言し、FastAPIはその宣言を使って受信データを自動的に検証し、そこからドキュメントを生成する。
- **自動生成されるインタラクティブなAPIドキュメント。** FastAPIは、別途ドキュメントを書くステップを必要とせず、コードから直接（OpenAPI標準経由で）閲覧可能でインタラクティブなAPIドキュメントページを生成する。
- **最初からビルトインの非同期サポート。** FastAPIはPythonの `async`/`await` 構文を中心に設計されており、大量の同時進行するI/Oバウンドな作業（データベースや外部APIの待機）を効率的に処理する必要のあるAPIに自然にフィットする。

## どちらを選ぶか

- **Flask**は、最大限のシンプルさとコントロールが欲しいとき、あるいはFastAPIの追加の構造が必要ない小さなものを構築しているときに適している。
- **FastAPI**は、自動バリデーション、自動生成ドキュメント、標準で備わった非同期パフォーマンスの恩恵を受けられる「本格的な」APIを構築しているときに適している——これが、特に他のサービスや、明確なAPIドキュメントを求めるフロントエンドチームから利用される、新しいPythonのAPIプロジェクトにおける一般的なデフォルトになっている理由だ。

どちらも大きなフルスタックフレームワークよりはるかに軽量であり続けており、両方とも、プロジェクトが実際に必要とする部品だけを追加できる*からこそ*よく選ばれている。
