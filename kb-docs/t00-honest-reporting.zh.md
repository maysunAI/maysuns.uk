# 诚实汇报完成状态——不假装"做完了"

## 解决的问题

当一个任务里有一部分确实做不到位——没有后端能验证支付、没有真实凭证、某个功能只在模拟数据下能跑——很容易产生一种冲动，想把它"做得看起来"已经完成了：代码能编译、界面能渲染、演示看起来能用。"看起来完成"和"真的能用"之间的差距，往往要等到有人真的依赖它、结果失败了才会暴露出来——而且经常是在最糟糕的时刻暴露，比如一个真实客户的付款在悄无声息中根本没被验证过。

## 规则

当一个任务里有一部分在当前条件下技术上做不到或做不完整时，不能用看起来完整、但实际不生效或有安全漏洞的代码/文档把这一点掩盖过去。而是要做到：

1. **能做的部分真正做实**——不偷工减料。
2. **明确标出做不到的部分**——用代码注释、界面上的警示条、文档里的⚠️小节——用足够醒目的方式，不会被忽略。
3. **说清楚具体差什么**——不是简单说"暂不支持"，而是具体列出缺的东西（比如"需要后端来验证Stripe的webhook""需要真实的用户认证系统+数据库"）。
4. **主动提前说，不是被追问才承认**——汇报完成情况的时候就要说清楚这个缺口，不是等别人发现问题了才承认。

## 为什么这条这么重要

如果某样东西看起来安全好用，但实际不是——比如一个没有真实验证机制的支付流程——它的失败方式不是"稍微有点失望"，而是"有人真的在生产环境里依赖了它，结果吃了亏"。一句诚实的"这部分还不是真的"，永远比一句不诚实的"这看起来能用"更有价值。

## 最常适用的场景

- 支付、多用户账号、权限校验——任何需要真实后端或真实凭证才能真正安全的功能
- 任何只在模拟数据下测试过、没有在真实使用条件下验证过的情况
- 任何一部分依赖提出需求的人目前还没有的资源（真实API Key、真实服务器、真实商业账号）的任务
