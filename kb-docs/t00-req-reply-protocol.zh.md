# req/reply 沟通协议

## 解决的问题

长时间的AI协作工作会话——横跨数小时、数天，甚至多个并行项目——存在一个上下文问题：实质性的回答被打进聊天窗口，随后滚动消失，之后几乎无法再找到。如果你问「三天前我们关于X决定了什么」，诚实的答案通常是「在滚动记录里的某个地方，祝你好运」。

req/reply协议是一个简单的修复方案：把「*终端*」和「*记录*」分开。

## 工作原理

1. **请求写入持久化日志。** 问题、指示、粘贴的资料（包括值得保留的命令行输出），会被写入一个持续累积的请求文件，而不是只在会滚动消失的聊天窗口里说一遍。
2. **实质性的回答写入持久化的回复日志**，而不只是留在终端里。每一个真正的答案——一个决定、一段说明、一段生成的内容——都会以统一、可搜索的格式写入回复文件。
3. **终端本身保持简洁。** 助手不会在聊天窗口里重复完整答案，而是给出一句简短确认（例如「回复已更新，第N节」），并指向日志所在位置。这样既让实际内容保存在持久的地方，也让实时对话保持可读。
4. **每一条回复都能追溯到出处。** 回复会注明它回答的是请求中的哪几行，这样几个月后依然能够重建「当初为什么这么做」，而不用靠猜。
5. **只追加，不覆盖（append-only）。** 日志中的旧条目永远不会被原地改写或删除，新的上下文以带日期的新区块形式追加。这样保留下来的是一份如实的历史，而不是被篡改过的。

## 为什么重要

- **可搜索性。** 几周前做的决定只需一次搜索就能找到，而不是埋没在可能连所用工具本身都不一定会保留的聊天记录里。
- **上下文能延续。** 会话可以在冷启动状态下被重新接续——无论是同一个人，还是全新的AI会话——都不需要从头把一切重新解释一遍。
- **可追责性。** 因为日志只能追加且带有日期，它就相当于一份审计记录，记录了问了什么、答了什么、什么时候答的。
- **终端噪音少。** 保持实时对话简洁，意味着你不用为了找到下一行可执行的内容，而不断滚动过一段段重复的内容。

## 如何自己应用这套方法

尝试这个方法不需要任何特殊工具——一个纯文本文件（或者两个，一个记入站请求，一个记出站回答）就足够了。核心纪律很简单：把内容写在某个持久的地方、让实时对话保持简短、绝不悄悄覆盖之前写下的内容。
