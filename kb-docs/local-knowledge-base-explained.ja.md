# ローカルナレッジベース（RAG + LangChain）とは何か

## 解決する課題

言語モデルは訓練されたことしか知らない。あなたの非公開の文書、会社の社内Wiki、訓練カットオフ以降に公開されたものについて、その素材を直接手渡さない限り、モデルは信頼できる情報を持っていない。文書が変わるたびにモデルを再訓練するのは現実的ではない。ローカルナレッジベースは、モデルを再訓練することなく、*あなた自身の*特定のコンテンツについて正確で最新の回答をモデルに与えるための方法だ。

## 核となる技術：RAG（検索拡張生成）

RAGは、記憶だけに頼るのではなく、モデルを検索ステップと組み合わせる。

1. あなたの質問は、**埋め込み（embeddings）**（意味を数値で表現したもの）を使ったベクトル類似検索によって、ナレッジベースと比較される——正確なキーワードを共有するものだけでなく、意味的に関連するチャンクを見つける。
2. 最も関連性の高いチャンクが取得される。
3. それらのチャンクは、あなたの質問と*一緒に*モデルに渡され、あいまいな記憶からではなく、目の前にある実際の出典素材をもとに回答が生成される。

回答が取得されたテキストに基づいているため、RAGシステムはしばしばそれがどの一節から来たのかを引用できる——「信じてください」を「これが出典です、確認してください」に変える。これは、事実に関する質問でのハルシネーションを減らすうえで大きな意味を持つ。

## 一般的なツール群：LangChain

RAGパイプラインを手作業で構築するということは、同じ配管作業——文書のチャンク分割、ベクトルストアの管理、取得したコンテキストのフォーマット、会話メモリの処理——を何度も解決することを意味する。LangChainは、これらを再利用可能な構成要素（ドキュメントローダー、テキストスプリッター、ベクトルストア連携、メモリモジュール）としてパッケージ化したオープンソースフレームワークであり、すべての層をゼロから書く必要がない。

パイプラインに複数の可動部分（検索 → 推論 → ツール呼び出し → 応答）がある場合、そのオーバーヘッドに見合う価値が出てくる。単純な単発のプロンプト応答呼び出しに対しては、モデルAPIを直接呼ぶ方がシンプルなことが多い——フレームワークは自動的に優れているわけではなく、トレードオフなのだ。

## その他の一般的なツール群：LlamaIndexとLangGraph

**LlamaIndex**は、課題のもう半分——「自分のデータを取り込み、検索可能にする」——に特化している：データソース（ファイル、データベース、API）に接続し、チャンク分割・インデックス化し、効率的にクエリする。検索そのものが主に必要で、汎用のオーケストレーションフレームワークまでは必要ない場合、LangChainより狭いが、しばしばよりシンプルな選択肢になる。

**LangGraph**（LangChainチームが開発）は別の課題に取り組む：アプリケーションが単純な一直線のパイプライン以上のもの——分岐するロジック、ループ、前のステップに戻ることがある複数ステップ、複数の協調するエージェント——を必要とするようになると、単純なチェーンでは足りなくなる。LangGraphはアプリケーションを「明示的な状態を持つステップのグラフ」としてモデル化し、こうしたより複雑で循環的、あるいはマルチエージェントなフローを、関数呼び出しの連鎖よりも推論・デバッグしやすくする。単純なRAG（検索→回答）だけでは足りず、検索の上に本当の意味での多段階のエージェント的な振る舞いが必要になったときに使われることが多い。

この4つをおおまかに位置づけると：RAGは*技術*そのもの、LlamaIndexはそれを実装する際の「データ取り込み／検索」の半分に特化したもの、LangChainは汎用的な*パイプライン*構築の部品を提供するもの、そしてLangGraphはそのパイプラインが一直線ではなく分岐・ループ・複数エージェントを必要とするようになったときの*オーケストレーション*を担うものだ。

## 「ローカル」というのは具体的にどういうことか

これをすべて自分のマシン上で実行すること——クラウドAPIの代わりに、ローカルのベクトルストアと（例えばOllama経由の）ローカルで動くモデルを使うこと——は、あなたの文書が自分のデバイスから外に出ないことを意味する。トレードオフは通常、プライバシーとオフラインでの利用可能性と引き換えに、ホスト型モデルと比べた能力と速度になる。

## どこに当てはまるか

- 文書と対話するツール（Wiki、PDF、コードベース）
- 特定の製品のドキュメントから回答しなければならないサポートボット
- 汎用的な流暢さよりも、*特定の、変化し続ける*知識体系に対する正確さの方が重要なあらゆるケース

これは万能のアップグレードではない。外部知識に依存しないタスク（創作的な文章、一般的な推論）では、検索はあまり利益をもたらさず、オーバーヘッドだけが増える。
