# 設計文書——「何をしたか」ではなく「なぜそうしたか」を記録する

## 解決する課題

要件定義書は「何を作るか」を記録し、概要書は「何ができたか」を記録する。しかしどちらも「ある特定の決定が、なぜそのような形になったのか」は記録しない——特に、一見すると簡略化・整理できそうに見えるが、実は過去に発見したバグや制約のせいであえてそう書かれている決定については。この記録がないと、後から（人であれAIであれ）「親切心で」コードを整理し直した結果、避けようとしていたそのバグに逆戻りしてしまうことがある。

## 基本の考え方

設計文書は、本当にトレードオフを伴った決定だけを記録し続ける、追記式のログだ——コードが何をするかを言い換えるのではなく、外部の読み手（あるいは未来の自分自身）がコードを読むだけでは復元できない推論を記録する：

- 試したが却下されたアプローチと、その理由
- コード自体からは明らかでない制約
- 苦労して見つけたバグと、実際に何が直したのか（単に「ここは壊れていた」ではなく）

## 書くべきでないこと

コードを見れば明らかなことは書かなくていい。実装を読んだレビュアーが助けなしに理由を理解できるなら、エントリは不要だ——コードの一行一行を文章で言い換えるだけの設計文書は、本当に価値のある情報がノイズに埋もれてしまい、かえって役に立たなくなる。

## なぜ重要か

「複雑に見えるものを簡略化したい」という本能は、たいてい正しい——ただし、その複雑さがひと目ではわからない理由で存在している場合を除いて。設計文書は、二度目にそのコードに触れる人がこの判断を正しく下せるようにするものだ：同じバグをもう一度発見し直す代わりに、これがすでにシンプルな方法で試され、却下されたことがあるかどうかを確認できる。

## 使い方

本当にトレードオフを伴う決定がなされたとき、あるいは分かりにくいバグの根本原因が特定されたときに、短いエントリ（決定・理由・日付）を追記する——日常的な通常業務では記録しない。実際に関連する変更をする前に読み返せるくらい、小さく保たれているべきで、コミット履歴の二番目のコピーになってはいけない。
