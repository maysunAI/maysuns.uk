# AI支援によるコード変更のための信頼レベル

## 解決する課題

AIアシスタントがコードベースに広範な編集権限を持っていると、間違った場所で「親切すぎる」振る舞いをしてしまいやすい。実は意図的に残されていた回避策を「修正」してしまったり、触ると危険だからこそわざと粗いままにしてあったファイルをリファクタリングしてしまったり、人間が特に手を入れて調整したものを書き換えてしまったりする。この解決策は、AIをあらゆる場所で読み取り専用に制限することではない（それではAIを使う意味がなくなる）。そうではなく、コードの領域ごとに、どれくらいの慎重さで臨むべきかを伝えることだ。

## この慣習

関連するコードの近くに置く短いコメントやメモという形の、シンプルなタグ付けの仕組みで、いくつかの信頼レベルのいずれかを示す。

- **高信頼（High trust）** — 機微であったり、根幹に関わったり、静かに壊れやすい部分。AIはここでは慎重であるべきで、変更する前に、何をなぜ変更する予定なのかを説明し、後から報告するだけで済ませてはいけない。
- **中信頼（Medium trust）** — ほとんどのコードにとって妥当なデフォルト。通常の変更は問題なく、特別な対応は不要。
- **低信頼（Low trust）** — 大胆に編集して問題ない領域。使い捨てファイル、生成された出力、使い捨ての足場コードなど、慎重になることが何のメリットもなく作業を遅らせるだけの箇所。
- **触ってはいけない（Do-not-touch）** — 明示的なマーカー（例えば「手動修正済み」というメモ）で、人間が理由があってこれを手作業で編集したことを意味し、人間が別途指示しない限り完全にそのままにしておくべきもの。

## なぜ重要なのか

- **静かな破壊を防ぐ。** 何のシグナルもなければ、AIアシスタントはすべてのコードを同じように変更して安全なものとして扱う——つまり、正当な理由があってわざと雑に書かれていた唯一のファイルが「きれいに整理」されてしまい、何かが壊れることになる。
- **すべてを遅くもしない。** 逆の失敗モード——*すべて*を高リスクとして扱い、あらゆる編集の確認を求めること——は、AIアシスタントを一緒に作業していて煩わしい存在にしてしまう。段階分けすることで、慎重さを一律に適用するのではなく、実際のリスクに応じて調整できる。
- **意図を明示的かつ持ち運び可能にする。** 信頼レベルのタグはセッションをまたいでも、異なるAIツールをまたいでも生き残る——それはコードの中のただのメモであって、先週の会話の記憶の中にしか存在しないものではない。

## 自分なりに応用するには

2〜3段階のレベルを選び（複雑な仕組みは不要）、それぞれに短いマーカーを決め、実際に重要な箇所——重要な設定、手作業で調整されたアルゴリズム、「親切な」書き換えが本当の損害をもたらしかねない箇所——に適用する。それ以外はタグなしのままにしておき、通常の慎重さをデフォルトとすればよい。
