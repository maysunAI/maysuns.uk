# Git 基础

## 它是什么

Git是一个分布式版本控制系统：它随时间追踪一组文件的变化，让你可以保存带名字的检查点、精确查看何时改了什么、在不打扰主线工作的情况下分出一个分支去尝试新东西，并把变更重新合并回去——包括多人在同一批文件上所做的修改。

「分布式（distributed）」意味着仓库的每一份拷贝都拥有完整的历史记录，而不仅仅是指向中央服务器的一个指针——这正是Git能够离线工作的原因，也是为什么像GitHub这样的服务只是建立在Git之上的一种托管便利,而不是Git本身运行所必需的东西。

## 核心概念

- **仓库（Repository）** ——Git正在追踪的一个项目文件夹，包含它的完整历史。
- **提交（Commit）** ——一个已保存的检查点：那一刻文件的快照，附带一条说明改了什么、为什么改的消息。
- **分支（Branch）** ——一条独立的开发线。默认分支（通常叫 `main`）通常是「当前的工作版本」；新建一个分支可以让你在准备好合并回去之前，自由实验或开发某个功能而不影响它。
- **合并（Merge）** ——把一个分支的变更整合进另一个分支。
- **远程仓库（Remote）** ——托管在别处（例如GitHub）的仓库副本，用于备份和协作。`push` 把你的提交发送过去；`pull` 把别人的提交拉取下来。
- **差异（Diff）** ——一个文件两个版本之间逐行的实际差异，这是你（或者审查者）查看某次提交究竟改了什么的方式。

## 为什么它是默认的起点

- **可逆性。** 几乎任何操作都可以撤销——一次糟糕的改动、一次意外的删除、一次没成功的实验——因为历史记录被完整保留了下来。
- **协作时不会互相踩踏。** 分支让多个人（或者一个AI助手和一个人）能在同一个代码库上工作，而不会直接覆盖彼此正在进行中的工作。
- **一份真正的审计记录。** 「改了什么、什么时候改的、为什么改」在几个月后依然可以被回答,而不只是靠某人模糊的记忆。

## 一套最简单的日常工作流

```
git status          # what's changed since the last commit
git add <files>      # stage specific changes to include
git commit -m "..."  # save a checkpoint with a description
git push              # send commits to the remote (e.g. GitHub)
git log --oneline    # see recent history
```

仅凭这些——status、add、commit、push——就能覆盖日常Git使用中的绝大部分场景。当你开始同时处理不止一件事情，或者需要和别人协作时，分支、合并和变基（rebase）才会变得重要起来。
