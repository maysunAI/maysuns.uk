# 回答する前に、依頼を整理してマッピングする

## 解決する課題

AIアシスタントへの実際の依頼が、きれいに一つにまとまった質問であることはめったにない。多くの場合、修正・新しい依頼が2つ・脱線した話題・貼り付けられた背景情報など、いくつものことが一度に、思いついた順のまま書き込まれる。そのまま素直に答えようとすると、AIアシスタントは一番目立つ部分だけを拾い、残りを静かに取りこぼしがちだ。依頼した本人も、何かが漏れたことにずっと気づかないことすらある。

## 基本の考え方

回答する前に、届いた依頼を番号付きのリストに整理し直し、それぞれが元の文章のどの部分に対応するかを明示する。このマッピングができてから初めて、実際の回答に入る——同じ順序で、番号ごとに一つずつ答えていく。

これは文章を読みやすく言い換えるためのものではない。「複数の依頼が入り混じった一段落」を、回答に取りかかる前に「N個の、それぞれ独立して答えられる項目」に変換する、構造的なチェックポイントだ。こうすることで、何かが静かに未対応のまま置き去りにされることがなくなる。

## 実際にはどうするか

1. まず届いたメッセージ全体を最後まで読み、すぐには答えない。
2. それを番号付きの項目に分ける——修正、新しい依頼、質問、確認など、実際にいくつあるかに応じて、決まった数ではない。
3. 各項目について、どの番号に対する回答かを明示しながら、はっきりと答える。
4. ある項目に答えるためにさらに情報が必要だとわかった場合は、それも独立した項目として明示する——黙って省略しない。

## なぜ重要か——これは「AIとの正しい話し方」でもある

このパターンは、AIアシスタントがどう応答すべきかという話だけではない。双方向の習慣だ。人間側の実践的なバージョンで言えば：依頼に複数の部分がある場合、5つの異なる依頼を一つの密な段落に詰め込むのではなく、実際に分けて書く（番号付きリストにする、異なる依頼の間に空行を入れる、あるいは単に「もう一つあります」と言葉で区切る）ことが役に立つ。入力の時点で異なる要点の境界がはっきりしているほど、出力の時点でそのどれかが失われる可能性は低くなる。

これが最も重要になるのは、まさに間違えやすい場面だ：長いセッション、修正と新しい作業が一つのメッセージに混在している場合、あるいは回答が複数回に分かれて届く場合（バックグラウンドAgent、複数パートに分かれた返信）——こうした状況では、「すべての部分が実際に対応されたかどうか」が、一目でわかることではなくなってしまう。

## T00の他の部分との関係

これはT00のreq/replyプロトコルの土台にある規律だ：永続的なログが依頼と回答を残し続ける一方で、「マッピング」こそが、そのログの中の密度の高い1件のエントリが、実際に回答される前に一部を静かに失ってしまうことを防いでいる。
