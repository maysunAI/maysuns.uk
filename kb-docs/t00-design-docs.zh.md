# 设计文档——记录「为什么」，不只是「做了什么」

## 解决的问题

需求文档记录要做什么，交付概览记录做出了什么。但两者都不记录「某个具体决定为什么是这么做的」——尤其是那些乍一看好像可以简化/清理掉，但其实是因为踩过某个坑或遇到某个约束才特意写成这样的决定。没有这份记录，后面无论是人还是AI再来改一遍，都可能"好心办坏事"地把代码"优化"回当初想避开的那个坑。

## 核心思路

设计文档是一份持续追加的记录，只记那些真正涉及取舍的决定——不是复述代码做了什么，而是记录一个外部读者（或者未来的自己）光看代码本身还原不出来的那部分推理：

- 试过但被否决的方案，以及为什么否决
- 代码本身看不出来的约束条件
- 踩过的坑，以及真正解决它的方法是什么（不只是说"这里以前坏过"）

## 什么不该写进去

代码本身已经看得出来的东西不用写。如果一个人读实现代码就能看懂这么做的理由，不需要专门记一条——一份把每行代码都用文字复述一遍的设计文档反而没用，因为真正有价值的信息会被淹没在噪音里。

## 为什么重要

看到一段代码"看起来写复杂了"就想简化，这种直觉通常是对的——除非这份复杂度是有原因的，而这个原因光看一眼代码看不出来。设计文档正是让这种判断在第二次经手时能做对的关键：先查一下这是不是已经试过简单做法、被否决了，而不是重新踩一遍同一个坑。

## 怎么用

每当出现一次真正的取舍决定，或者一个不容易看出来的坑被真正找到根因，就追加一条简短记录——决定/理由/日期，日常的常规工作不需要记。它应该保持精简到"改动前真的会去翻一下"的程度，而不是变成commit历史的第二份拷贝。
