# req/reply コミュニケーションプロトコル

## 解決する課題

AIを使った作業セッションが数時間、数日、あるいは複数プロジェクトにまたがって長く続くと、コンテキストの問題が生じる。実質的な回答はチャットウィンドウに打ち込まれ、スクロールで流れていき、後からほぼ見つけられなくなる。「3日前にXについて何を決めたっけ」と聞かれたとき、正直な答えはたいてい「スクロールバックのどこか、頑張って探して」になってしまう。

req/replyプロトコルはこれに対するシンプルな解決策だ。「*端末（ターミナル）*」と「*記録*」を分離する。

## 仕組み

1. **リクエストは永続的なログに書き込む。** 質問、指示、貼り付けた資料（保存する価値のあるコマンドライン出力を含む）は、流れて消えるチャットプロンプトに話しかけるだけでなく、継続的なリクエストファイルに書き込まれる。
2. **実質的な回答は端末だけでなく、永続的な返信ログに書き込む。** 決定事項、説明、生成されたコンテンツといった実際の回答は、一貫した検索可能なフォーマットで返信ファイルに記録される。
3. **端末自体は簡潔さを保つ。** チャットウィンドウに回答全文を繰り返す代わりに、アシスタントは短い確認（「返信を更新しました、セクションN」）を返し、ログを指し示す。これにより、実際のコンテンツは永続的などこかに存在しつつ、リアルタイムの会話は読みやすく保たれる。
4. **すべての返信ブロックは元の出典まで辿れる。** 返信には、リクエストのどの行に対する回答なのかが記載される。これにより、数ヶ月後でも「なぜこうしたのか」を推測せずに再構築できる。
5. **追記のみ（append-only）。** ログ内の古いエントリはその場で書き換えたり削除したりしない。新しいコンテキストは、日付付きの新しいブロックとして追記される。これにより、改ざんされていない正直な履歴が保たれる。

## なぜ重要なのか

- **検索性。** 何週間も前に下した決定も、使ったツールが保存すらしていないかもしれないチャット履歴に埋もれるのではなく、一回の検索で見つかる。
- **コンテキストの継続性。** セッションは、同じ人でも、新しいAIセッションでも、最初から全部を説明し直すことなく、途中から冷えた状態で再開できる。
- **説明責任。** ログが追記のみで日付付きであるため、何が聞かれ、何が答えられ、いつだったのかという監査証跡として機能する。
- **端末のノイズが少ない。** ライブチャットを簡潔に保つことで、次に実行すべき行を探すために繰り返しのコンテンツの段落をスクロールし続ける必要がなくなる。

## 自分なりに応用するには

これを試すのに特別なツールは必要ない。プレーンテキストファイル（あるいは2つ、入力用リクエストと出力用の回答を分けてもよい）で十分だ。核となる規律はシンプルで、「どこか永続的な場所に書き留めること」「ライブの会話は短く保つこと」「以前書いたものを黙って上書きしないこと」の3つだけだ。
