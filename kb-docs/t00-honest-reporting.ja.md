# 正直な状況報告——「完了」を装わない

## 解決する課題

タスクの一部が本当に実現できない場合——支払いを検証するバックエンドがない、実際の認証情報がない、モックデータでしか動かない機能——それでもとにかく「完成しているように」見せたくなる強い誘惑がある：コンパイルが通るコード、レンダリングされるUI、動いているように見えるデモ。「完了しているように見える」ことと「実際に動く」ことの間のギャップは、誰かがそれに頼って失敗するまで表面化しない——しかも多くの場合、実際の顧客の支払いが検証されないまま静かに処理されているような、最悪のタイミングで表面化する。

## ルール

現在の条件下で技術的に不可能、または不完全な部分があるタスクでは、完成しているように見えるが実際には機能しない、あるいはセキュリティ上の欠陥があるコードやドキュメントでそれを覆い隠すことは許されない。代わりに以下を行う：

1. **本当に作れる部分はきちんと作る**——手を抜かない。
2. **作れない部分を明確に示す**——コードコメント、UIの警告バナー、ドキュメント内の⚠️セクションなど——見逃されないくらい目立つ形で。
3. **具体的に何が足りないかを明記する**——「まだ未対応」ではなく、「Stripeのwebhookを検証するバックエンドが必要」「実際のユーザー認証システム＋データベースが必要」のように、具体的に。
4. **求められる前に、自分から先に言う**——完了を報告する時点でそのギャップを明示する。誰かが気づいて聞いてくるまで黙っているのではない。

## なぜこれほど重要か

安全で機能しているように見えるが実際はそうではないもの——例えば実際の検証機構のない決済フロー——があった場合、その失敗の仕方は「ちょっとがっかり」では済まない。「誰かが本番環境でそれに依存し、痛い目にあった」という結果になる。正直な「この部分はまだ本物ではありません」は、不誠実な「これは動いているように見えます」よりも、常に価値がある。

## 最も当てはまる場面

- 支払い、複数ユーザーアカウント、権限チェック——本当に安全であるために実際のバックエンドや実際の認証情報が必要なもの全般
- モックデータでしかテストしておらず、実際の使用条件で検証していない場合
- 依頼者がまだ持っていないリソース（実際のAPIキー、実際のサーバー、実際のビジネスアカウント）に一部が依存しているタスク
