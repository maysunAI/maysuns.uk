# Gitの基本

## それが何であるか

Gitは分散型バージョン管理システムだ。一連のファイルへの変更を時間とともに追跡し、名前付きのチェックポイントを保存し、何がいつ変更されたかを正確に確認し、メインの作業ラインを乱すことなく何かを試すために枝分かれし、変更を再び統合できる——同じファイルに取り組む複数人による変更も含めて。

「分散型」とは、リポジトリのすべてのコピーが中央サーバーへのポインタだけでなく完全な履歴を持つことを意味する——だからこそGitはオフラインで動作し、GitHubのようなサービスはGit自体に必須のものではなく、その上に乗ったホスティングの利便性にすぎない。

## 中核となる概念

- **リポジトリ（Repository）** — Gitが追跡しているプロジェクトフォルダで、その完全な履歴を含む。
- **コミット（Commit）** — 保存されたチェックポイント。その瞬間のファイルのスナップショットに、何がなぜ変更されたかを説明するメッセージが添えられたもの。
- **ブランチ（Branch）** — 独立した開発のライン。デフォルトブランチ（多くの場合 `main`）は通常「現在の作業版」であり、新しいブランチを作ることで、マージして戻す準備ができるまで触れずに実験したり機能を構築したりできる。
- **マージ（Merge）** — あるブランチの変更を別のブランチに統合すること。
- **リモート（Remote）** — 別の場所（例えばGitHub）でホストされているリポジトリのコピーで、バックアップと共同作業に使われる。`push` はコミットをそこへ送り、`pull` は他の人のコミットを取り込む。
- **差分（Diff）** — ファイルの2つのバージョン間の実際の行単位の差異で、これによって（あなた自身やレビュアーが）あるコミットが正確に何を変更したかを確認できる。

## デフォルトの出発点である理由

- **可逆性。** 悪い変更、うっかりした削除、うまくいかなかった実験など、ほぼ何でも元に戻せる。履歴が保存されているからだ。
- **互いの邪魔をしない共同作業。** ブランチによって、複数の人（あるいはAIアシスタントと人間）が、互いの進行中の作業を直接上書きすることなく、同じコードベースで作業できる。
- **本物の監査証跡。** 「何が、いつ、なぜ変更されたか」は、誰かがうろ覚えで思い出すことではなく、何ヶ月も後でも答えられる。

## 最小限の日常ワークフロー

```
git status          # what's changed since the last commit
git add <files>      # stage specific changes to include
git commit -m "..."  # save a checkpoint with a description
git push              # send commits to the remote (e.g. GitHub)
git log --oneline    # see recent history
```

これだけで——status、add、commit、push——日常的なGit利用の大部分をカバーする。ブランチ作成、マージ、リベースは、常に複数のことに同時に取り組んだり、他の人と共同作業したりするようになってから関係してくる。
