T00 Knowledge Base

The working system behind every project on this site

T00 is a personal AI-collaboration framework.

At its core, T00 defines a communication protocol, memory rules, and project organization conventions.

It's a real, evolving personal system, not a polished framework for others to install.

It isn't one rulebook — it's a mix of a communication protocol, memory/context conventions, and a growing library of how-to guides, all written in plain Markdown.

What's inside

Communication protocol

A structured request/reply format so context survives long sessions and several projects running in parallel.

Project organization conventions

A consistent folder and file structure so dozens of projects stay findable instead of turning into a junk drawer.

Trust-level tagging

Code and docs are labeled by how freely they can be changed — from "safe to edit freely" to "don't touch without asking" — so AI assistance doesn't silently break things that matter.

A documentation habit

How-to guides written in plain Markdown, growing over time as the system changes.

Five T00 concepts, explained

These describe the general mechanism, not this site's private working files — each one is a reusable idea you could adopt in your own AI workflow.

The req/reply communication protocol

A convention for keeping AI collaboration organized: substantive answers get written into a persistent reply log rather than only spoken in the terminal, while the terminal itself stays terse — short confirmations only. That way a session spanning hours or days can be picked back up without re-reading a wall of scrollback.

Why it matters for T00: it's the backbone that keeps long, multi-project sessions traceable instead of scattered across chat history.

Read the full guide (.md) →

Trust levels for AI-assisted changes

A high / medium / low tagging convention that tells an AI assistant how freely it can change something. High-trust areas need a heads-up before changes; low-trust areas can be edited boldly; untagged code defaults to a sensible middle ground.

Why it matters for T00: it stops an eager AI from silently rewriting something that was deliberately built a certain way.

Read the full guide (.md) →

Persistent memory across sessions

The idea of giving an AI assistant a small, durable memory file — preferences, conventions, recurring feedback — that carries forward between separate conversations, so context doesn't need to be re-explained from scratch every time.

Deliberately minimal: not a full transcript, just the handful of durable facts worth remembering.

Read the full guide (.md) →

Custom slash commands

Packaging a repeatable multi-step workflow — like "estimate this task" or "publish this build" — into a single reusable command instead of re-typing the same instructions every time.

Turns a recurring multi-paragraph instruction into a one-word trigger, keeping repetitive workflows consistent.

Read the full guide (.md) →

Skills: on-demand instruction sets

A way of packaging specialized instructions so an AI assistant only loads them when actually relevant, instead of every conversation carrying the full weight of every capability it might ever need.

Keeps day-to-day interactions lighter while still making deep, specialized knowledge available the moment it's needed.
